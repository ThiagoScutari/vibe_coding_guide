📁 Iniciar o Repositório

Fluxo para o seu projeto pipeline-dev-software.html — do zero ao GitHub com main branch configurada.

01 Configurar identidade do Git (só na primeira vez) setup ▶

O Git precisa saber quem fez cada commit. Configure uma vez na máquina.

TERMINAL
$ git config --global user.name "Seu Nome"
$ git config --global user.email "seu@email.com"
$ git config --global init.defaultBranch main # define main como padrão
02 Inicializar o repositório na pasta do projeto início ▶

Vá até a pasta onde está o pipeline-dev-software.html e inicialize o Git.

TERMINAL
$ cd /caminho/para/sua/pasta # entre na pasta do projeto
$ git init # cria o repositório local
$ git status # confirma que funcionou
03 Criar .gitignore e fazer o primeiro commit commit ▶

O .gitignore diz ao Git o que não versionar (arquivos temporários, senhas, node_modules). Depois fazemos o primeiro commit na main.

TERMINAL — PRIMEIRO COMMIT
$ # cria .gitignore básico
$ echo ".DS_Store\n.env\nnode_modules/\n*.log" > .gitignore
$ git add . # adiciona TODOS os arquivos ao stage
$ git commit -m "feat: initial commit — pipeline dev software guide"
$ git log --oneline # veja o commit criado
Convenção de mensagens de commit (Conventional Commits):
feat: nova feature  ·  fix: correção de bug  ·  docs: documentação  ·  style: formatação  ·  refactor: refatoração  ·  chore: manutenção
02

🌿 Trabalhando com Branches

A main é sagrada — código que vai para produção. Todo trabalho novo acontece em uma feature branch.

FLUXO VISUAL DE BRANCHES
← main (sempre estável)
← feature/minha-branch
← main após merge ✓
01 Criar uma nova branch para trabalhar branch ▶

Sempre crie uma branch a partir da main atualizada. Use nomes descritivos com prefixo.

CRIAR E ENTRAR NA BRANCH
$ git checkout main # sempre parta da main
$ git pull # garanta que está atualizado
$ git checkout -b feature/adicionar-secao-akita
$ # ou a forma mais moderna:
$ git switch -c feature/adicionar-secao-akita
Prefixos de branch recomendados:
feature/ nova funcionalidade  ·  fix/ correção de bug  ·  docs/ documentação  ·  refactor/ refatoração  ·  hotfix/ correção urgente em produção
02 Ver e navegar entre branches navegação ▶
GERENCIAR BRANCHES
$ git branch # lista branches locais (* = atual)
$ git branch -a # lista locais + remotas
$ git switch main # volta para main
$ git switch feature/minha-branch # muda para outra branch
$ git branch -d feature/branch-antiga # deleta branch local (já mergeada)
03

💾 Fazendo Commits

FLUXO COMPLETO DE COMMIT
$ git status # veja o que mudou
$ git diff # veja as mudanças em detalhe
$ git add pipeline-dev-software.html # adiciona arquivo específico
$ # ou adiciona tudo:
$ git add .
$ git status # confirme o que está no stage (verde)
$ git commit -m "feat: adicionar secao lições do mundo real (Akita)"
Working Directory
arquivos modificados
→
Staging Area
git add
→
Repository
git commit
→
Remote (GitHub)
git push
DESFAZER MUDANÇAS ANTES DO COMMIT
$ # tirar arquivo do stage (mas manter as mudanças no arquivo):
$ git restore --staged arquivo.html
$ # descartar todas as mudanças NÃO commitadas (CUIDADO — irreversível):
$ git restore .
$ # guardar mudanças temporariamente (stash) para trocar de branch:
$ git stash # salva e limpa working dir
$ git stash pop # recupera as mudanças salvas
CORRIGIR O ÚLTIMO COMMIT (antes do push)
$ # esqueceu um arquivo? adicione e emende o commit:
$ git add arquivo-esquecido.html
$ git commit --amend --no-edit # mantém a mensagem anterior
$ # corrigir apenas a mensagem do último commit:
$ git commit --amend -m "feat: mensagem corrigida"
Nunca use --amend em commits já enviados ao GitHub (push). Isso reescreve o histórico e causa conflitos para outras pessoas (ou para você mesmo em outra máquina).
04

🔀 Merge — Unindo Branches

01 Merge da feature branch na main merge ▶

Quando a feature está pronta e testada, mergeia na main. Sempre faça o merge estando na branch de destino (main).

MERGE COMPLETO
$ git switch main # vá para a branch destino
$ git pull # atualize a main antes
$ git merge --no-ff feature/adicionar-secao-akita
$ # --no-ff: cria um commit de merge explícito (histórico mais claro)
$ git log --oneline --graph # veja o merge no histórico
$ git push # envie para o GitHub
$ git branch -d feature/adicionar-secao-akita # limpa branch local
02 Resolver conflitos de merge conflito ▶

Conflitos acontecem quando a mesma linha foi editada em duas branches. O Git marca o arquivo com os dois lados — você escolhe qual fica.

O ARQUIVO COM CONFLITO VAI FICAR ASSIM
<<<<<<< HEAD (sua versão — main)
<h2>Versão original do título</h2>
=======
<h2>Título editado na feature branch</h2>
>>>>>>> feature/minha-branch
RESOLVER E CONCLUIR O MERGE
# 1. Abra o arquivo, edite manualmente e remova as marcações <<<, ===, >>>
$ git add arquivo-com-conflito.html # marque como resolvido
$ git commit # conclui o merge

⏪ Rollback — Voltando no Tempo

Existem 4 formas de desfazer coisas no Git, cada uma para uma situação. Escolha errada pode causar mais problemas.

git revert — Mais seguro, use sempre que possível
Cria um novo commit que desfaz as mudanças. Não reescreve o histórico. Seguro para branches públicas.
REVERT (SEGURO)
$ git log --oneline # veja os commits para pegar o hash
a3f8c12 feat: adicionar secao rollback
b2d1e45 fix: corrigir typo no titulo ← quero desfazer este
c9a7f88 feat: initial commit
$ git revert b2d1e45 # desfaz aquele commit específico
$ git push # envia o commit de revert
git reset --soft — Desfaz o commit, mantém as mudanças no stage
Volta o ponteiro do commit mas mantém as alterações prontas para commitar novamente. Útil para reagrupar commits.
RESET SOFT (SÓ LOCAL)
$ git reset --soft HEAD~1 # desfaz 1 commit, mantém no stage
$ git reset --soft HEAD~3 # desfaz os últimos 3 commits
git reset --mixed — Desfaz o commit e retira do stage (padrão)
Volta as mudanças para o working directory. As edições ficam lá, mas fora do stage.
RESET MIXED (SÓ LOCAL)
$ git reset HEAD~1 # --mixed é o padrão
git reset --hard — ⚠️ Apaga tudo, irreversível
Desfaz o commit E descarta as mudanças dos arquivos. Use com cuidado — os dados são perdidos.
RESET HARD — CUIDADO
$ git reset --hard HEAD~1 # perde as mudanças do último commit
$ git reset --hard c9a7f88 # volta para aquele commit específico
$ # se já fez push e quer forçar (NUNCA na main compartilhada):
$ git push --force-with-lease # mais seguro que --force
Use --hard + force-push apenas em branches de feature pessoais, nunca na main ou em branches compartilhadas. Isso reescreve o histórico e causa conflitos sérios.

Quando usar cada um?

Comando Arquivos modificados Histórico Uso ideal
revert mantém preserva + novo commit Desfazer em produção / main. Sempre preferível.
reset --soft mantém (no stage) reescreve (local) Reagrupar commits antes do push.
reset --mixed mantém (fora do stage) reescreve (local) Recomeçar o staging de um commit.
reset --hard PERDE reescreve (local) Descartar tudo de uma feature branch local.

🌐 Subindo para o GitHub

01 Criar repositório no GitHub e conectar remoto ▶

Crie um repositório vazio no GitHub (sem README), depois conecte o repositório local a ele.

CONECTAR AO GITHUB
$ # após criar o repo no github.com, copie a URL e rode:
$ git remote add origin https://github.com/seuuser/seu-repo.git
$ git push -u origin main # primeiro push + configura tracking
$ # de agora em diante, basta:
$ git push
02 Push de uma feature branch push ▶
PUSH DE FEATURE BRANCH
$ git switch feature/minha-feature
$ # primeiro push da branch:
$ git push -u origin feature/minha-feature
$ # pushes subsequentes:
$ git push
